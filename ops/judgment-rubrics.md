# ops/judgment-rubrics.md — 判斷力外化

> 目的：把高階模型擅長的「拿捏」轉成任何等級模型可執行的判準。
> 用法：拿不定主意時查對應 Rubric，逐條核對判準。判準成立就照做，不要憑感覺推翻。
> 每條 Rubric 附一正例一反例。反例代表「看起來像但不是」的情境——那是最容易誤判的地方。

---

## Rubric 1：何時升級模型

**升級判準（任一成立即升級）**：
- 同一任務已失敗 2 次（驗收不過，或兩次答案方向相反）。
- 任務屬於「一次做錯的修復成本 > 10 倍模型差價」：架構選型、資料 schema 變更、對外發布內容、涉及金額/法遵的計算。
- 任務需要跨 ≥3 個模組的全局一致性設計。

**不升級判準**：只是任務量大（量大 → 拆小派多個便宜 agent，不是升級）；只是想「更保險」但無上述訊號。

- 正例：Sonnet 兩次重構 payment 模組都讓測試紅在不同地方 → 停止重試，opus 重派 + 附兩次失敗摘要。
- 反例：要改 200 個檔案的 import 路徑，覺得「量大所以用 opus」→ 錯。這是機械式批改，拆批派 haiku，驗收跑 build。

## Rubric 2：何時算真的完成

**完成 = 以下全部成立**：
1. 驗收條件（派工時寫的那份）逐條有證據：測試輸出、實跑結果、或 read-back 確認。
2. 證據來自 fresh-context 驗收，不是實作者自評（見 `ops/model-dispatch.md` 第 6 條）。
3. 沒有留下 `[未驗證]`、TODO、或「應該可以」字樣；有就明列為已知限制。

**不算完成**：程式碼寫完但沒跑過；文件寫完但沒 read-back；「測試大部分過了」（紅一個就是沒完成，除非使用者明示接受）。

- 正例：「已完成：測試 42/42 綠（輸出貼於 test-log.txt），fresh agent read-back 確認 README 三步驟可照做。」
- 反例：「已完成，程式碼邏輯我檢查過沒問題。」——沒有機器證據，退回。

## Rubric 3：何時停下來問使用者

**必問（任一成立）**：
- 決策不可逆或成本高：刪資料、對外發送、花錢、改動 git history、變更治理檔的硬規則。
- 兩個合理解讀會導致 >30% 的工作量差異或互斥的架構方向。
- 發現任務前提錯誤（使用者以為 X 存在，實際不存在）。

**不問，自己決定並明說假設（任一成立）**：
- 任何解讀做出來的東西可以低成本改向（命名、檔案結構、範例選擇）。
- 屬於既定慣例範圍（語言選擇、格式偏好——CLAUDE.md 環境事實已定義）。

- 正例：使用者說「清掉舊的 migration」——刪檔不可逆且「舊的」邊界模糊 → 列出打算刪的清單問確認。
- 反例：使用者說「寫個 script 整理 log」，沒說輸出格式 → 不要問，選 JSON lines 做出來並在回報第一行寫「假設輸出 JSONL，要換格式一行改」。

## Rubric 4：什麼訊號代表方向錯了，該換路而非重試

**換路訊號（任一成立就停止當前路線）**：
- 每修一個問題冒出一個新問題，連續 3 輪（打地鼠模式）。
- 為了讓方案成立，開始需要繞過或修改原本的驗收條件。
- 修復程式碼行數已超過重寫該單元的預估行數。
- 需要引入與現有架構慣例衝突的依賴或模式才能繼續。

**換路動作**：停手 → 3 行總結「試了什麼、卡在哪、為什麼判定死路」→ 回到上一個決策點列 ≥2 條替代路線 → 屬 Rubric 3 必問範圍就問使用者，否則選一條並明說理由。

- 正例：為了讓某 library 相容，已 patch 三處內部行為還在冒錯 → 換路：評估換 library 或自寫薄封裝。
- 反例：測試紅了一次就想砍掉重練 → 錯。單次失敗先讀錯誤訊息修復；換路訊號是「模式」不是「單一事件」。

## Rubric 5：品質底線怎麼驗（模糊題與品味題）

**可機器驗的先機器驗**：格式、連結有效性、測試、lint、數字加總——這些永遠不靠模型自評。

**只能靠判斷的（文案品味、架構優雅度、命名品質）——誠實條款：這是 harness 補不了的**。降低風險的做法按序：
1. 多答案評審：派 2–3 個 agent 各出一版，fresh opus 評審選優並說明取捨。這提升期望品質，但**上限仍是評審模型的品味**。
2. 具體化 rubric：把「好」翻成可核對的清單（例如文案：結論先行？每段一個論點？無空洞形容詞？）。
3. 以上都不夠時，**明說做不到**：「這題的品味判斷超出我可靠範圍，建議：升級最強可用模型重做／找人類第二意見」。假裝有把握是最壞選項。

- 正例：對外 announcement 文案 → 三版並行 + opus 評審 + 明告使用者「語氣拿捏建議你最後過目」。
- 反例：模型自己給自己的文案打 9/10 然後直接交付。自評分數無資訊量，禁止作為品質證據。
