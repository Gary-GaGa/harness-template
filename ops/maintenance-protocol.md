# ops/maintenance-protocol.md — 治理檔維護協議

> 讀者：任何想修改 CLAUDE.md 或 ops/*.md 的未來 session。
> 原則：制度可以演化，但演化必須留痕、可回退、且不放大權限。

---

## 1. 權限分級

**可自行修改（改完在回報中告知使用者即可）**：
- `ops/LESSONS.md`：append 新教訓（格式見第 3 節）。只准 append，不准改寫舊條目。
- CLAUDE.md「環境事實」節：填入/更新客觀事實（build 指令、目錄結構）。
- ops 檔中的明顯錯誤：壞掉的路徑、過時的工具名、錯字——修正時必須是「對齊現實」而非「改變規則」。

**動之前必須先問使用者（Rubric 3 必問範圍）**：
- CLAUDE.md 的硬規則：新增、刪除、放寬任何一條。
- `ops/model-dispatch.md` 的閾值（200 行、失敗 2 次升級等）與「驗證不自驗」原則。
- `ops/judgment-rubrics.md` 的任何判準。
- 新增 hook、修改 settings.json 權限、新增 subagent 定義檔。
- 刪除任何治理檔或其中任何一節。

**永遠不准做**：
- 為了讓當前任務過關而臨時放寬規則（「這次先跳過驗收」）。規則擋路時的正確動作：照規則做，或問使用者。
- 修改本檔第 1 節（權限分級本身）——只有使用者能改。

## 2. 修改程序（適用所有治理檔）

1. `cp <檔> <檔>.bak-$(date +%Y%m%d)`（CLAUDE.md 硬規則 3）。
2. 改動。
3. 在檔案頭部的變更記錄加一行：`日期 | 改了什麼（一句話）| 觸發原因`。無變更記錄區的檔案，第一次修改時建立。
4. 派 fresh-context agent 用 T5 模板（`ops/delegation-templates.md`）read-back 驗收：新規則是否與既有規則衝突、措辭是否會被 Haiku 級模型誤讀。
5. 回報使用者：diff 摘要 + 為什麼。

## 3. 教訓回寫：ops/LESSONS.md

格式定義以 `ops/LESSONS.md` 檔頭註解為唯一真源（模板已內建該檔）。不要在本檔或其他檔重複定義格式。

**寫入時機**：返工 ≥2 輪的錯誤、驗收抓到的實作盲點、任何「早知道就好了」時刻。session 收尾時寫，不要中途分心。

**寫法判準**：教訓必須是可執行的祈使句。正例：「改 Xcode project 設定後必跑 clean build 再驗，增量 build 會假綠。」反例：「要更小心 build 設定。」——不可執行，等於沒寫。

## 4. 精簡閾值（防止制度自身膨脹）

- `LESSONS.md` 超過 **30 條**：整理出重複模式，向使用者提案「哪幾條該升格為規則、哪幾條已過時可歸檔到 LESSONS-archive.md」。升格與歸檔都需使用者核准。標 `[通用]` 的條目另提案上繳模板 repo（跨專案共用），同樣需使用者核准。
- CLAUDE.md 超過 **120 行**：立即精簡——長內容抽成 ops 引用檔，CLAUDE.md 留路由一行。
- 任何 ops 檔超過 **200 行**：檢查是否混入了該落在別檔的內容，拆分提案給使用者。
- 每季（或使用者要求時）跑一次全檔健檢：派 fresh agent 用 T5 找規則衝突、失效路徑、從未被引用的規則（從未用到的規則是候選刪除項）。

## 5. 回退

任何規則改動後若連續 2 個 session 造成明顯摩擦（頻繁誤觸、產出變慢且無品質收益），回退到 .bak 版本，並在 LESSONS.md 記錄該次制度實驗的失敗原因。
