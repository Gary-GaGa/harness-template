# ops/delegation-templates.md — 派工 Prompt 模板

> 用法：主對話派 subagent 時，複製對應模板、填空、顯式指定 model（規則見 `ops/model-dispatch.md` 第 4 條）。
> `{{ }}` 為填空。「回報合約」段每個模板都有，不可刪——那是主 context 的防洪堤。

---

## 通用回報合約（每個派工 prompt 結尾附上）

```
回報規則：
- 只回結論、檔案:行號、落檔路徑。長內容（>50 行）寫檔傳路徑 + ≤10 行摘要。
- 禁止貼原始碼全文、網頁原文、完整工具輸出。
- 找不到/跑不動/不確定 → 直說 + 列出已嘗試步驟。不要編造。
```

## T1 搜尋／盤點（model: haiku）

```
目標：找出 {{要找什麼，例如：所有讀取環境變數 DATABASE_URL 的位置}}。
動機：{{為什麼找，例如：準備把連線設定集中到 config 模組，需要完整影響面}}。
範圍：{{目錄或檔案類型限定}}。
驗收條件：
- 覆蓋 {{範圍}} 內全部命中，寧可多報待我過濾，不可漏報。
- 每筆含 檔案:行號 + 一句話上下文。
回報格式：清單，每筆一行 `路徑:行號 — 上下文`，按目錄排序，上限 {{40}} 行；超過則落檔 {{路徑}} 並回摘要。
[附通用回報合約]
```

## T2 實作（model: sonnet）

```
目標：實作 {{功能}}，位於 {{檔案/模組}}。
動機：{{使用情境與呼叫方，subagent 靠這個決定介面設計取捨}}。
規格：{{輸入/輸出/edge cases，逐條}}。
限制：遵循 {{既有慣例檔或範例檔路徑}} 的風格；不引入新依賴，除非先在回報中提出理由。
驗收條件（交付前自查）：
- {{測試指令}} 全綠；新邏輯有對應測試，至少含 {{正常路徑 + 兩個 edge cases}}。
- 無 TODO / 無 [未驗證] 殘留。
回報格式：改動檔案清單（路徑:行號範圍 + 一句話）、測試輸出末 10 行、已知限制（無則寫「無」）。
[附通用回報合約]
```

## T3 重構（model: sonnet；跨 ≥3 模組的設計先升 opus 出方案）

```
目標：重構 {{範圍}}，目的是 {{可讀性/去重複/解耦…擇一具體寫}}。
動機：{{觸發原因，例如：此模組近三次修改都引發回歸}}。
硬限制：外部行為不變——重構前先確認 {{測試指令}} 全綠並記錄基線；重構後同一套測試必須同樣全綠。基線紅的測試先回報，不要順手修（那是另一個任務）。
驗收條件：
- 測試前後皆綠，且未修改任何測試斷言（改了要逐條說明理由）。
- {{量化目標，例如：重複區塊從 4 處收斂為 1 處}}。
回報格式：基線測試結果、重構後測試結果、結構變化摘要（≤10 行）、風險點清單。
[附通用回報合約]
```

## T4 研究（model: sonnet；結論影響架構決策時第二意見用 opus）

```
目標：回答 {{具體問題，避免「調查一下 X」這種開放題}}。
動機：{{這個答案將用來決定什麼}}。
方法要求：優先官方文件/原始來源；每個關鍵結論附來源 URL 與日期；來源互相矛盾時並列不裁決。
驗收條件：
- 每條結論可追溯到來源；無來源的推測明確標 [推測]。
- 回答「不知道/查不到」是合格輸出，編造是不合格輸出。
回報格式：結論先行（≤5 條）、每條附來源、最後一段列「未能查證的部分」。全文落檔 {{路徑}}，主對話只回結論段。
[附通用回報合約]
```

## T5 審查／驗收（fresh-context 必須；model 按 `ops/model-dispatch.md` 第 6 條選）

```
你是驗收者，沒有參與實作，這是刻意的——不要向實作方查證，獨立判斷。
受驗物：{{路徑或 diff 範圍}}。
原始驗收條件：{{逐條貼上派工時的驗收條件}}。
執行：
1. {{檔案類 → 「讀取後用自己的話回答：這份文件要求讀者做什麼？」；程式碼類 → 「跑 {{測試指令}}，貼結果末 10 行」}}
2. 逐條核對驗收條件，每條標 PASS/FAIL + 證據（檔案:行號 或 測試輸出行）。
3. 額外檢查：hardcoded secrets、與 {{慣例檔}} 的衝突、規則自相矛盾處。
回報格式：總判定（PASS / FAIL）+ 逐條表格 + FAIL 項的最小修復建議（只建議，不動手改）。
[附通用回報合約]
```
